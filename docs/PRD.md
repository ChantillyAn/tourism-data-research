## 1. 产品概述

### 1.1 一句话定义

文旅从业者的每日情报站——政策、数据、趋势，一个公众号全覆盖。

### 1.2 产品背景

中国文旅数据市场存在一个巨大的中间真空：

- **重量级产品**（百度慧眼、阿里云文旅）：年费数十万，面向省级文旅厅和5A景区
- **免费散装信息**：散落在31个省级文旅厅网站、OTA媒体稿、行业资讯中，需要手动跑几十个网站去收集
- **中间地带**：完全空白。没有一个面向个人从业者和中小企业的文旅数据聚合产品

本产品填补这个空白。

### 1.3 核心价值主张

本产品卖的不是数据本身（数据都是公开的），卖的是三件事：

1. **时间** — 用户不需要每天跑31个政府网站 + 49个公众号，我们替他跑
2. **筛选** — 每天产生的几百条信息中，我们只挑出最有价值的5-8条
3. **解读** — 不仅告诉你"发生了什么"，还告诉你"这对你意味着什么"（周报/月报）

如果用户自己做这件事，每天至少需要30-60分钟。我们把这个时间压缩到1分钟（读日报）。

### 1.4 产品形态

信息订阅产品（日报 + 周报 + 月报），不是SaaS平台。

| 载体 | 角色 | 说明 |
|------|------|------|
| 微信公众号 | 核心载体 | 用户每天打开微信就能看到，零门槛 |
| 静态网站 | 辅助载体 | SEO获客 + 历史归档 + 搜索功能 |
| 小红书 | 引流渠道 | 数据可视化卡片引流 |
| 抖音 | 引流渠道（低优先级） | 短视频品牌曝光 |

### 1.5 项目决策历程

本产品经过5轮压力测试迭代：

| 版本 | 方向 | 压力测试评分 | 致命问题 |
|------|------|:---:|------|
| v1 | 文旅社媒数据分析平台 | 2/10 | 数据源API不可用；免费替代品够用 |
| v2 | AI文旅创作助手 | 4/10 | PWA不适合中国；AI推荐天花板低 |
| v3 | 文旅内容AI工作台 | — | 差异化不足（通用LLM可替代）；AIGC备案障碍 |
| v4 | 文旅数据简报（日/周/月报） | 通过 | 收入预期需校准；坚持6个月是最大考验 |

关键转折点：创始人指出"脚本生成、标题生成、文案生成这些功能，用户完全可以在自己的AI ChatBot里面完成"——真正稀缺的不是AI能力，而是**数据聚合**。

## 2. 目标用户

### 2.1 主要用户画像

#### 画像A：景区中层管理者

| 维度 | 描述 |
|------|------|
| **身份** | 4A/5A景区的市场部、运营部中层（主管/经理级别） |
| **年龄** | 28-45岁 |
| **核心需求** | 行业趋势、竞品景区动态、政策信息、写汇报材料用的数据 |
| **当前痛点** | 公司不会买百度慧眼这种重量级产品，自己去几十个网站翻数据太耗时 |
| **信息获取现状** | 零散地看几个行业公众号，偶尔在百度上搜，主要靠同事传来的截图和群里的分享 |
| **付费意愿** | 中等。如果内容确实有用，5-20元/篇能接受，但不愿意承诺月付 |
| **使用场景** | 早上通勤时快速浏览日报；写周报/月报时搜索历史数据；向领导汇报时引用权威数据 |

#### 画像B：酒店/民宿经营者

| 维度 | 描述 |
|------|------|
| **身份** | 中小酒店老板、民宿主（自营或小团队经营） |
| **年龄** | 30-50岁 |
| **核心需求** | 本地市场竞争情报、客源地变化趋势、OTA政策变化、政府补贴信息 |
| **当前痛点** | 知道"数据很重要"但不知道去哪里找、找到了也不知道怎么解读 |
| **信息获取现状** | 主要靠OTA后台的经营数据，行业信息几乎为零 |
| **付费意愿** | 偏低。更看重"有没有直接能用的信息"，对分析类内容兴趣一般 |
| **使用场景** | 看到日报中"本地"相关的政策赶紧点进去；节假日前看客流预测数据调价格 |

#### 画像C：文旅创业者/规划师

| 维度 | 描述 |
|------|------|
| **身份** | 正在进入或已在文旅行业的创业者、旅游规划咨询公司从业者 |
| **年龄** | 25-40岁 |
| **核心需求** | 行业全景视野、案例参考、投资决策依据、写可行性报告用的行业数据 |
| **当前痛点** | 信息碎片化，缺乏系统性视野；买一份前瞻研究院的报告要几千元 |
| **信息获取现状** | 比较积极，已关注多个行业公众号，但信息仍然不系统 |
| **付费意愿** | 最高。愿意为高质量的系统性分析付费，月报对他们价值最大 |
| **使用场景** | 写商业计划书时需要行业数据支撑；向投资人汇报时引用权威来源；评估某个区域的文旅投资价值 |

#### 画像D：地方文旅局基层人员

| 维度 | 描述 |
|------|------|
| **身份** | 县级/区级文旅局工作人员 |
| **年龄** | 25-45岁 |
| **核心需求** | 对标数据（其他省份/城市的做法和数据）、政策文件汇总、写材料用的数据引用 |
| **当前痛点** | 需要全国对比数据来写报告，但只能找到本省的；上级要求学习外地经验但不知道去哪找案例 |
| **信息获取现状** | 主要看本省文旅厅的通知，很少主动关注其他省份 |
| **付费意愿** | 低（个人付费意愿低，但可能用于单位内部报销） |
| **使用场景** | 写年度总结时需要"全国文旅行业概况"段落；领导让"学习XX省的做法"时需要快速了解 |

### 2.2 用户不是谁

- 不是省级文旅厅（他们有百度慧眼等重量级工具）
- 不是文旅内容创作者/博主（他们的需求是内容灵感，不是行业数据）
- 不是纯游客/旅行爱好者
- 不是需要实时数据的用户（我们是日报，不是实时看板）
- 不是需要定制化数据报告的大企业（他们应该买百度慧眼或请咨询公司）

### 2.3 用户的信息消费习惯

| 习惯 | 对产品设计的影响 |
|------|---------------|
| 主要使用微信获取行业信息 | 公众号是唯一正确的主载体 |
| 在通勤/间隙时间快速浏览 | 日报必须短（200-300字），能在1分钟内读完 |
| 对付费内容的接受度在提升，但仍然偏低 | 日报免费降低门槛，周报/月报按篇付费降低承诺感 |
| 更信任"有观点的人"而不是"工具平台" | 产品要建立个人IP调性，不是冷冰冰的机器人 |
| 习惯在工作场景中使用（不是休闲场景） | 内容风格要专业、可引用，不能太随意 |
| 会把有用的文章转发给同事/领导 | 内容格式要适合转发，标题要一目了然 |

### 2.4 典型用户场景

**场景1：早高峰快速浏览**

> 张经理是贵州一家4A景区的市场部主管。早上7:50挤地铁时打开微信，看到"文旅早知道"的推送。花40秒看完今天的3条信息：四川新出了一个景区安全管理通知、携程发布了清明假期搜索数据、黄山开始实行分时预约。他截图了四川那条发到工作群："咱们也需要关注一下这个通知。"

**场景2：写报告时查数据**

> 李科长在县文旅局工作，需要写一份"2026年上半年本县文旅工作总结"。领导要求"对标先进，找差距"。他打开文旅数据简报的网站，搜索"浙江 文旅 2026"，找到了浙江今年上半年的几个关键数据（游客量同比增长、旅游收入、重点项目），写进报告里并标注了来源。

**场景3：周末深度阅读**

> 王老板经营着3家民宿。周一早上看到周报推送，标题是"本周深度：入境游144小时免签政策扩围对周边游市场的影响"。虽然要花5块钱，但这个话题和他直接相关（他的民宿在桂林，外国游客越来越多）。他付费阅读后觉得分析得不错，转发到了当地民宿老板的微信群。

**场景4：做投资决策**

> 陈总正在评估是否在海南投资一个文旅项目。他购买了文旅数据简报的月报，里面有海南近3个月的游客量、酒店入住率、新开业项目盘点。数据来源标注清晰（海南省文旅厅、携程研究院），可以直接引用到可行性报告中。

## 3. 产品目标

### 3.1 核心目标

1. 成为文旅从业者获取行业信息的**第一入口**
2. 通过信息聚合和深度分析建立**个人IP和行业权威性**
3. 验证文旅数据简报这个品类的**付费需求**

### 3.2 Phase 1 关键指标

Phase 1（前6个月）不追求收入，但需要关注以下指标来判断产品是否有价值：

| 指标 | 观测方式 | 健康信号 | 危险信号 |
|------|---------|---------|---------|
| 公众号关注数 | 公众号后台 | 稳定增长，6个月达到500+ | 3个月后仍不到100 |
| 日报打开率 | 公众号后台 | 稳定在15%以上 | 持续低于5% |
| 周报付费转化率 | 公众号后台 | 关注者中3%以上购买 | 持续低于1% |
| 内容被转发/引用 | 公众号后台 + 搜索 | 每周有转发到行业群 | 从未被转发 |
| 网站搜索量 | Google Analytics | 来自搜索引擎的自然流量持续增长 | 无搜索流量 |
| 用户留言/反馈 | 公众号留言 | 有用户主动提问、建议 | 零互动 |

**重要说明**：这些不是"必须达到的KPI"，而是"观测健康度的信号"。Phase 1的核心任务是跑通流程、验证内容质量。如果6个月后所有信号都是"危险"，则需要重新评估产品方向。

### 3.3 非目标（明确不做的事）

| 非目标 | 原因 |
|--------|------|
| 不做SaaS平台 | 不开发用户登录、数据看板等功能 |
| 不做实时数据 | 日报是每日一次，不是实时推送 |
| 不做社群运营 | 公众号为主，不建微信群（管理成本高，容易沉寂） |
| 不做视频内容分析 | 不涉及抖音/小红书的内容数据分析 |
| 不做任何预测/展望 | 不预测"下周/下月会怎样"，只报道已发生的事实和数据 |
| 不做UGC | 不开放用户投稿或评论互动（Phase 1） |
| 不做国际市场 | 只覆盖中国大陆文旅市场 |

## 4. 内容产品定义

### 4.1 日报

| 维度 | 定义 |
|------|------|
| **定位** | 用户每天的文旅行业"一分钟早知道" |
| **频率** | 每天1期，早8:00推送（含周末和节假日） |
| **篇幅** | 200-300字，可配1张信息卡片图 |
| **生产方式** | Python自动化脚本（Claude Code开发），调用LLM API处理 |
| **人工介入** | 每天5分钟快速浏览自动生成的内容，必要时微调 |
| **价格** | 免费 |
| **分发渠道** | 公众号推送 + 网站归档 |

**日报内容模板**：

```
【文旅早知道 · YYYY年MM月DD日】

📋 政策动态（0-3条）
- [省份/全国] 标题（一句话摘要）（来源：XX）

📊 数据快报（0-3条）
- 数据标题（关键数字 + 一句话解读）（来源：XX）

🔥 今日热点（0-2条）
- 热点事件标题（一句话概述）（来源：XX）

数据来源：各省文旅厅官网、行业媒体公众号
更多历史数据：[网站URL]
```

**日报内容来源**：

| 板块 | 数据来源 | 采集方式 |
|------|---------|---------|
| 政策动态 | 全国各省文旅厅官网、文旅部官网 | changedetection.io 监控网页变更 |
| 数据快报 | 携程/美团/马蜂窝的PR数据发布、行业研究报告 | 公众号内容采集（见第6章） |
| 今日热点 | 行业媒体（环球旅讯、品橙旅游等） | 公众号内容采集 + RSS |

**日报质量标准**：

| 标准 | 要求 | 不合格示例 |
|------|------|----------|
| 来源标注 | 每条信息必须标注来源 | "据悉，某景区..."（谁据悉？） |
| 政策核实 | 政策类信息必须可追溯到原文链接 | 转述二手消息而非原文 |
| 数据标注 | 数据必须标注来源和时间范围 | "游客增长30%"（什么时间段？和什么比？） |
| 不凑数 | 无新信息时发布"今日无重大更新" + 行业知识 | 强行把不相关的内容拼凑成日报 |
| 不预测 | 只报道已发生的事实 | "预计五一假期将迎来..."  |
| 篇幅控制 | 200-300字 | 超过400字（用户没耐心读完） |

### 4.2 周报

| 维度 | 定义 |
|------|------|
| **定位** | 本周文旅行业的趋势分析和深度解读 |
| **频率** | 每周1期，周一早上推送 |
| **篇幅** | 2000-3000字 |
| **生产方式** | AI生成初稿 + 人工编辑（补充观点、校验数据、调整表述） |
| **人工编辑量** | 预计1-2小时/期 |
| **价格** | 付费（公众号付费文章，3-5元/篇） |
| **免费预览** | 前200-300字免费可读，核心分析在付费墙后 |
| **分发渠道** | 公众号付费文章（完整版）+ 网站（摘要版 + 目录，引导付费阅读） |

**周报内容结构**：

```
【文旅周报 · 第X期 · YYYY.MM.DD-MM.DD】

一、本周政策要闻（3-5条，含解读）
   每条包括：
   - 政策名称 + 发布机构 + 日期
   - 核心内容（100字以内）
   - 影响分析：谁会受影响？影响有多大？
   - 行动建议：相关从业者应该怎么应对？

二、数据观察（2-3个数据点）
   每个数据点包括：
   - 数据来源 + 统计口径 + 时间范围
   - 关键数字 + 同比/环比变化
   - 趋势解读：这个数字说明了什么？

三、行业事件深度分析（1-2个）
   每个事件包括：
   - 事件背景和经过
   - 多角度分析（对不同类型从业者的影响可能不同）
   - 对从业者的启示（实操层面，不是泛泛的"要关注"）
```

**周报质量标准**：

| 标准 | 要求 |
|------|------|
| 原创分析 | 必须包含原创分析观点，不是纯信息搬运 |
| 实操性 | 政策解读必须说明"对谁有影响、应该怎么做"，不能只说"值得关注" |
| 数据规范 | 数据必须标注来源、时间范围、统计口径 |
| 多角度 | 同一事件对景区、酒店、民宿、旅行社的影响可能不同，需分别说明 |
| 付费价值 | 读者付了3-5元，必须觉得"值了"——标准是"读完后知道了至少1件不知道的事" |

### 4.3 月报

| 维度 | 定义 |
|------|------|
| **定位** | 月度行业全景复盘，可作为决策参考和汇报素材 |
| **频率** | 每月1期，次月第一周发布 |
| **篇幅** | 5000-8000字，含图表 |
| **生产方式** | 人工主导 + AI辅助数据处理和初稿 |
| **人工工作量** | 预计3-5天/期（含数据收集、分析、写作、制图） |
| **价格** | 付费（公众号付费文章，9.9-19.9元/篇） |
| **免费预览** | 前300-500字 + 目录免费，完整内容付费 |
| **分发渠道** | 公众号付费文章（完整版）+ 网站（摘要版） |

**月报内容结构**：

```
【文旅月报 · YYYY年M月】

一、月度数据总览
   - 核心指标仪表盘（图表）：游客量、旅游收入、酒店入住率、机票均价等
   - 同比/环比变化分析
   - 本月数据亮点和异常值

二、政策月度盘点
   - 本月国家级政策汇总表（发布日期、名称、核心内容、影响领域）
   - 本月地方政策要点（按区域分类）
   - 政策趋势判断：本月政策释放了什么信号？

三、区域热点分析（2-3个区域）
   - 选取本月表现突出或出现异常的区域
   - 深入分析原因（政策驱动？事件驱动？季节性？）
   - 对其他区域的参考价值

四、深度专题（1个）
   - 每期选1个主题做深度研究
   - 包含数据、案例、分析、对从业者的启示
   - 格式适合打印和转发（部分用户会用于内部汇报）
```

**月报深度专题选题库**（前6期参考）：

| 期数 | 建议选题 | 选题理由 |
|------|---------|---------|
| 第1期 | 2026年中国文旅行业数据全景 | 开篇定调，展示产品的数据覆盖能力 |
| 第2期 | 民宿行业价格战：数据透视 | 民宿用户画像B高度关注 |

### 4.5 编辑准则与品牌调性

#### 品牌人格

本产品不是一个冷冰冰的数据推送机器人，而是"一个懂行的同事每天帮你整理行业信息"。

| 维度 | 是 | 不是 |
|------|---|------|
| 语气 | 专业、简洁、有信息量 | 官方公文腔、学术论文腔 |
| 立场 | 中立客观，基于数据说话 | 带立场的评论员、推销某个观点 |
| 深度 | 提供"发生了什么"+"意味着什么" | 仅转发原文、或过度分析 |
| 价值 | 帮读者省时间、提供可直接引用的信息 | 制造焦虑、贩卖情绪 |

#### 写作风格指南

**用词原则**：

| 原则 | 示例 |
|------|------|
| 用具体数字代替模糊形容 | ✅ "同比增长23%" ✗ "大幅增长" |
| 一句话一个信息点 | ✅ "海南3月接待游客580万人次" ✗ 长复合句 |
| 标注所有数据来源 | ✅ "（来源：携程研究院）" ✗ "据报道" |
| 区分事实和分析 | ✅ 事实段落 + 分析段落分开 ✗ 事实中夹杂主观判断 |
| 不使用标题党话术 | ✅ "黄山实行分时预约" ✗ "黄山放大招！这个变化影响所有景区" |
| 不做预测 | ✅ "清明搜索量同比涨42%" ✗ "清明假期将迎来旅游热潮" |

**引用规范**：

| 情况 | 引用方式 |
|------|---------|
| 政府政策 | 标注"来源：XX省文旅厅官网"，附原文链接（网站版） |
| OTA数据 | 标注"来源：携程研究院"、标注数据时间范围 |
| 行业媒体 | 标注"来源：环球旅讯"，不大段摘录原文 |
| 统计数据 | 必须标注统计口径（如"旅游收入"是旅游总收入还是旅游外汇收入） |
| 二手信息 | 必须追溯一手来源，不引用"据XX报道"的转引 |

**禁止事项**：

- 禁止发布未经核实的信息（宁可少一条，不可错一条）
- 禁止使用"据悉""据了解""有消息称"等模糊来源
- 禁止对政策进行过度解读（只分析写在纸面上的内容）
- 禁止做任何预测性表述
- 禁止抄袭（AI摘要必须改写为自己的表述，不能直接用原文句子）
- 禁止使用emoji表情代替正文（板块标题的emoji除外）

### 4.6 内容日历

文旅行业有明确的季节节奏。内容选题和工作强度应跟随行业周期：

| 时段 | 行业节奏 | 内容策略 |
|------|---------|---------|
| **1月** | 元旦数据出炉；春运启动；各省发布上年度数据 | 重点关注年度数据汇总 |
| **2月** | 春节黄金周（最大假期）；春节后各地复工 | 节前：预订数据；节后：春节数据复盘 |
| **3月** | 全国两会（文旅政策风向）；春季旅游启动 | 两会文旅相关提案/报告；春游数据 |
| **4月** | 清明假期；各省Q1数据出炉 | 清明数据；Q1行业回顾 |
| **5月** | 五一黄金周（重要假期）；中国旅游日(5.19) | 五一数据深度复盘（月报重点） |
| **6月** | 端午假期；暑期旺季前预热 | 端午数据；暑期趋势前瞻数据 |
| **7-8月** | 暑期旺季（年度高峰之一） | 密集关注各地客流、价格、热点事件 |
| **9月** | 中秋假期；暑期收尾数据 | 暑期数据复盘；中秋数据 |
| **10月** | 国庆黄金周（第二大假期）；Q3数据出炉 | 国庆数据深度复盘（月报重点） |
| **11月** | 淡季开始；各地秋冬旅游营销 | 冰雪旅游、温泉旅游专题 |
| **12月** | 年度总结密集期；冰雪旅游旺季 | 年度数据盘点；冰雪旅游数据 |

#### 特殊内容规划

| 类型 | 触发条件 | 内容形式 |
|------|---------|---------|
| 假期数据速报 | 每个法定假期结束后1-3天 | 日报加长版或特别版，汇总各来源的假期数据 |
| 突发事件 | 行业重大事件（如景区安全事故、重大政策发布） | 当日日报重点报道，下周周报深度分析 |
| 半年报/年报 | 每年7月和1月 | 月报特别版，篇幅可扩展到1万字 |
| 年度盘点 | 每年12月最后一周 | 年度特别版：十大政策、十大数据、十大事件 |

### 4.7 异常情况处理

| 异常情况 | 处理方式 |
|---------|---------|
| **当天无有价值信息** | 发布"今日无重大更新" + 行业小知识/历史回顾，绝不凑数 |
| **信息量爆发**（如重大政策日） | 日报可扩展到400-500字，但仍只保留最重要的5-8条 |
| **发布了错误信息** | 当天发布更正声明，说明错误内容和更正内容；网站版同步更新；建立更正记录 |
| **采集系统故障** | 启用人工降级方案：15分钟手动浏览关键公众号，手动编写日报 |
| **重大突发事件** | 如果事件在推送后发生（早8:00后），不做额外推送，在次日日报中报道 |
| **运营者休假/生病** | 日报可全自动运行（允许2-3天不做人工审核），周报/月报提前1期准备buffer |
| **节假日** | 照常推送日报（自动化不休息），周报如遇节假日可顺延1天 |

#### 公众号采集技术方案

**核心挑战**：微信公众号是设计上不开放的封闭系统，官方API只能操作自己的账号，无法读取他人公众号内容。

**可选方案对比**：

| 方案 | 自动化程度 | 稳定性 | 合规性 | 推荐度 |
|------|:---:|:---:|:---:|:---:|
| 今天看啥（jintiankansha.me）VIP + RSS | 中高 | 中高 | 灰色 | ★★★★ |
| Wechat2RSS 托管服务 | 中高 | 中 | 灰色 | ★★★ |
| WeWeRSS 自部署 | 中 | 低中（需频繁扫码） | 灰色 | ★★ |
| 人工浏览 + 脚本处理 | 低 | 高 | 合规 | ★★★ |

**推荐方案（分层）**：

1. **内容发现**：使用"今天看啥"VIP订阅目标公众号，生成RSS源
2. **内容获取**：通过RSS获取新文章URL，Python脚本抓取文章正文
3. **内容处理**：调用LLM API做摘要、分类、关键词提取
4. **人工兜底**：RSS方案失效时，每天10-15分钟人工浏览关键公众号

**法律风险说明**：已有商业化爬取公众号数据被判赔60万-300万的案例。本产品以原创分析为主体，引用公开信息时标注来源，不大段摘录原文。

## 7. 产品能力：记忆与自我更新

### 7.1 记忆系统

系统在每次产出日报/周报后，应积累以下"记忆"：

| 记忆类型 | 内容 | 用途 | 实现优先级 |
|---------|------|------|:---:|
| **已报道事件库** | 每条已报道的政策/事件/数据点 | 去重，避免重复报道同一事件 | Phase 1 |
| **话题热度追踪** | 某个话题被多少来源提及、持续多少天 | 识别持续性热点 vs 一次性新闻 | Phase 1 |
| **省份活跃度** | 各省份被提及的频率和数据产出频率 | 发现数据公开度高的省份，优化监控策略 | Phase 1 |
| **来源质量评分** | 每个数据源的更新频率、内容质量、采集成功率 | 优化数据源权重，淘汰低质量源 | Phase 1 |
| **用户关注模式** | 哪类内容打开率高、付费转化好 | 优化内容选题方向 | Phase 2 |

**记忆系统的实际作用示例**：

- 已报道事件库：3月14日报道了"文旅部发布玻璃栈道通知"，3月15日有3个来源再次提到这条政策→系统识别为已报道，不重复推送，但记录热度+1
- 话题热度追踪：发现"入境游"关键词连续10天被多个来源提及→标记为持续性热点，建议在周报中做深度分析
- 来源质量评分：发现某公众号连续2周没有产出与文旅相关的有价值内容→降低权重，减少采集频率

### 7.2 自我更新机制

| 机制 | 触发条件 | 行为 | 实现优先级 |
|------|---------|------|:---:|
| **来源权重调整** | 某来源连续7天无有价值内容 | 降低该来源的优先级 | Phase 1 |
| **新来源发现** | 现有来源中多次引用某个未监控的信息源 | 标记为"待添加监控"并通知运营者 | Phase 1 |
| **Prompt优化** | 每周统计AI摘要的人工修改率 | 如果某类内容频繁被人工修正，调整对应的Prompt | Phase 2 |
| **模板迭代** | 每月评估日报模板效果 | 基于积累数据调整模板结构 | Phase 2 |

## 8. 运营流程

### 8.1 日报生产SOP

```
每天执行（含周末和节假日）：

06:30  Python脚本自动触发
       ├── 拉取过去24小时的网页变更（changedetection.io）
       ├── 拉取公众号新文章（RSS源）
       ├── 去重（对比已报道事件库）
       ├── AI分类+摘要（DeepSeek API）
       └── 生成日报Markdown

07:00  日报草稿生成完毕
       └── 保存到公众号草稿箱 + 生成网站Markdown文件

07:00-07:30  运营者审核（5分钟）
       ├── 检查日报内容是否准确
       ├── 检查来源标注是否完整
       ├── 必要时微调措辞（不要改变事实）
       └── 确认可以发布

08:00  公众号推送
       └── 同步更新网站（git push触发Vercel构建）
```

**如果07:30前脚本未完成**：触发告警，运营者启动手动降级方案（15分钟手动编写日报）。

### 8.2 周报生产SOP

```
周日：
18:00  Python脚本自动触发
       ├── 聚合本周所有已处理数据
       ├── 查询持续性热点话题
       └── 调用Claude API生成初稿

周日晚 - 周一早：
运营者编辑周报初稿（1-2小时）
       ├── 校验所有数据和来源
       ├── 补充原创分析观点
       ├── 调整表述使其符合品牌调性
       ├── 设置付费分割点（前200-300字免费）
       └── 确认可以发布

周一 08:00  公众号推送（付费文章）
       └── 同步更新网站摘要版
```

### 8.3 月报生产SOP

```
每月最后一天：
18:00  Python脚本自动触发
       ├── 聚合当月所有数据
       ├── 生成月度统计（按类别、按省份）
       ├── 提取关键数据指标
       └── 调用Claude API生成大纲和初稿片段

次月第1-5天：
运营者写作月报（3-5天，每天1-2小时）
       ├── Day 1：基于大纲确定结构，收集补充数据
       ├── Day 2：完成"数据总览"和"政策盘点"章节
       ├── Day 3：完成"区域热点分析"章节
       ├── Day 4：完成"深度专题"章节
       ├── Day 5：制作图表、全文校对、排版
       └── 确认可以发布

次月第一周末  公众号推送（付费文章）
       └── 同步更新网站摘要版
```

### 8.4 内容纠错机制

| 错误类型 | 处理方式 | 时效 |
|---------|---------|------|
| **数据错误**（如数字写错） | 公众号发布更正声明（在下一期日报开头注明）；网站版直接修改并标注"已更正" | 发现后立即处理 |
| **来源错误**（如张冠李戴） | 同上 | 发现后立即处理 |
| **遗漏重要信息** | 在次日日报中补充报道，注明"昨日遗漏" | 次日 |
| **措辞不当** | 网站版直接修改；公众号无法修改已发布文章，在留言区置顶说明 | 发现后立即处理 |

**纠错原则**：宁可主动纠错损失一点面子，也不能让错误信息留在那里。读者信任一旦打破很难恢复。

### 8.5 运营者日常时间分配

| 任务 | 频率 | 时间 |
|------|------|------|
| 审核日报 | 每天 | 5分钟 |
| 编辑周报 | 每周 | 1-2小时 |
| 写作月报 | 每月 | 5-10小时（分散到5天） |
