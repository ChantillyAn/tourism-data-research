# 设计决策记录

> 记录产品设计过程中的关键决策及其依据。

---

## 决策1：产品方向从"SaaS平台"转向"信息简报"

**日期**：2026-02-19
**背景**：最初设计为"文旅社媒智能分析平台"（SaaS），经过3轮压力测试后放弃。
**原因**：
- 第三方数据API（飞瓜、灰豚）不对外开放，数据源前提不成立
- 公开数据质量天花板限制AI推荐价值
- "通用LLM+5分钟搜索"可替代核心功能
- SaaS开发周期长、技术风险高

**决策**：转向信息简报产品（日报+周报+月报），聚焦数据聚合价值而非AI生成。

---

## 决策2：目标用户从"文旅创作者"转向"文旅从业者"

**日期**：2026-02-19
**背景**：最初定位服务文旅内容创作者/MCN。
**原因**：
- 创作者的核心需求是"内容灵感"，通用LLM已能满足
- 创作者付费意愿和能力有限
- 从业者（景区管理者、酒店老板、文旅局人员）的数据需求更刚性

**决策**：目标用户改为文旅行业从业者。

---

## 决策3：日报采用AI全自动化

**日期**：2026-02-19
**背景**：压力测试指出"一人运营三频次内容"时间成本不可持续。
**原因**：
- 日报的核心价值是"每天出现"（留存工具），不是内容深度
- 200-300字的信息卡片，AI处理质量可接受
- Python脚本 + changedetection.io + LLM API 的技术方案成熟

**决策**：日报全自动化，人工只需每天5分钟快速检查。

## 决策4：触达渠道选择"公众号+网站+付费文章"

**日期**：2026-02-19
**背景**：评估了纯公众号、公众号+知识星球、公众号+微信群、公众号+网站等多种模式。
**原因**：
- 公众号是微信生态内最自然的订阅入口
- 网站提供SEO获客能力和历史归档
- 公众号付费文章零平台抽成
- 不建微信群——管理成本高，容易沉寂

**已知局限**：公众号付费文章是单篇购买非订阅制，Phase 2需迁移到订阅模式。

## 决策5：~~先选5-8个重点省份~~ → 全国31省全覆盖

**日期**：2026-02-19 → **2026-02-20 修订**
**背景**：最初设计为只覆盖5-8个重点省份，后经创始人反馈修订为全覆盖。
**原始原因（已废弃）**：
- ~~每个政府网站需要单独配置CSS选择器~~
- ~~政府网站改版频繁，维护负担大~~
- ~~5-8个重点文旅省份已覆盖大部分有价值信息~~

**修订原因**：
- 既然做公众号信息产品，就要覆盖全国所有省、自治区和直辖市
- changedetection.io 支持无限监控任务，技术上可行
- 采用分批上线策略（先高数据公开度省份，再逐步扩展）降低一次性调试压力

**决策**：覆盖全部31个省级行政区 + 文旅部（共32个监控目标），分3批上线。

## 决策6：Phase 1不追求收入

**日期**：2026-02-19
**背景**：压力测试显示500粉丝阶段付费文章月收入约120元，几乎无利润。
**原因**：
- 冷启动阶段核心任务是内容质量验证和用户积累
- 过早追求变现会扭曲内容方向（标题党、注水）
- 88元/月运营成本可承受

**决策**：Phase 1（0-6个月）定位为"验证期"，收入预期归零。6个月后根据数据决定是否进入Phase 2。

---

## 决策7：技术栈从 n8n 迁移至 Python脚本（Claude Code开发）

**日期**：2026-02-20
**背景**：最初技术方案采用 n8n（可视化工作流）+ DeepSeek + Coze/WeWeRSS。
**原因**：
- 创始人明确表示使用 Claude Code 开发 Python 脚本，不使用 Coze 或 n8n
- Python 脚本比可视化工作流更灵活，便于版本管理和调试
- WeWeRSS 稳定性不足（需频繁扫码），改用"今天看啥"VIP + RSS方案
- Python 生态丰富（feedparser、requests、Jinja2等），无需依赖中间层

**决策**：
- 工作流编排：n8n → Python脚本 + cron/systemd timer
- 公众号采集：WeWeRSS → 今天看啥VIP + RSS + Python脚本
- LLM调用：DeepSeek V3（日报） + Claude API（周报/月报）
- 开发工具：Claude Code

## 决策8：不做任何预测/展望

**日期**：2026-02-20
**背景**：最初PRD中周报包含"下周关注"、月报包含"下月展望"板块。
**原因**：
- 预测不仅没有意义，还会增加工作量
- 预测不准会产生负面反噬，损害信息产品的可信度
- 本产品的核心价值是"报道已发生的事实和数据"

**决策**：所有内容产品（日报/周报/月报）只报道已发生的事实和数据，不做任何预测。

## 决策9：增加 Phase 0（本地 Claude Code 驱动）作为冷启动阶段

**日期**：2026-02-20
**背景**：SPEC v0.2 设计了完整的服务器自动化流水线（腾讯云 + Docker + cron + changedetection.io + 今天看啥 VIP），但启动成本约 ¥125/月，且需要大量基础设施配置工作。
**原因**：
- 创始人提出先用 Claude Code 本地生成内容，跳过服务器和网站
- Phase 0 零额外成本（已有 Claude Code 订阅），可以立即开始
- 先验证内容质量和用户需求，再投入基础设施
- Claude Code 自身（Opus 4.6）的分析能力足以支撑日报分类和摘要

**决策**：
- 在 SPEC v0.2（Phase 1）之前增加 Phase 0 冷启动阶段
- Phase 0 仅做微信公众号，不做网站
- 使用 Claude Code 作为运行环境和 LLM 引擎
- 使用 wechat-article-publisher-skill（开源 Claude Code Skill）发布到公众号
- 数据采集简化为 RSS + WebSearch + 手动浏览
- Phase 0 稳定运行 1-2 个月后，根据数据决定是否升级到 Phase 1
- 详见 [Phase 0 实施方案](./phase0-implementation.md)

**已知局限**：
- 无自动化（每天需手动触发 Claude Code）
- 信息采集覆盖面低于 Phase 1（无 changedetection.io 全天候监控）
- 无 SEO 获客（无网站）

## 决策10：采用开源工具而非从零开发

**日期**：2026-02-20
**背景**：评估了多个开源项目（auto-news、FeedSummarizer、newsletter-LLM-o-matic 等），发现核心组件均有成熟开源方案。
**原因**：
- wechat-article-publisher-skill 直接解决公众号发布问题，无需自写微信 API 对接
- feedparser + BeautifulSoup4 是成熟的 RSS/HTML 处理库
- 自己编写的核心代码只需约 200 行（胶水代码），其余全用现成工具
- 不重复造轮子，集中精力在领域特色（文旅行业知识、编辑准则、内容质量）

**决策**：
- 公众号发布：wechat-article-publisher-skill（已安装）
- RSS 采集：feedparser（Python 库）
- HTML 解析：BeautifulSoup4（Python 库）
- 模板渲染：Jinja2（Python 库）
- 数据存储：SQLite（Python 内置）
- LLM 分析：Claude Code 自身（Phase 0）/ DeepSeek V3（Phase 1）
